# 现有系统分析报告

## 1. 当前项目基线

当前程序仍然以 `app.py` 为核心入口，是一个单文件 Streamlit 办公小程序，包含三块能力：

- PPT 生成
- Excel / CSV 表格生成与处理
- PPT 模板资源库入口

当前已完成的资源库骨架不是“下载模板”，而是“资源发现 + 元数据 + 索引 + 预览 + 评分”的基础层。初始化后，种子库已生成 **137 个资源端点**，其中包含 GitHub 仓库、站点集合页和 GitHub topic 索引页。

## 2. 现有 PPT 生成流程

当前 PPT 生成仍然是规则驱动：

1. 解析自然语言提示词
2. 粗略提取标题与章节
3. 生成页面大纲
4. 用固定主题色和固定版式渲染成 `.pptx`

这个流程能满足“快速出稿”，但本质上仍是“一个完整 deck 作为一个模板” 的思路，没有把 PPT 拆成可独立检索、可复用、可重组的页面级资源。

## 3. 当前结构的主要问题

### 3.1 模板粒度过粗

当前系统没有真正的页面级模板库，也没有组件级素材库。封面页、目录页、数据页、时间轴、对比页等仍然是写死的渲染逻辑，而不是从资源库里智能检索和组合。

### 3.2 缺少页面识别与拆解

没有把完整 PPT 拆成单页，并为每页建立类型、行业、场景、风格、颜色、内容结构等元数据。结果就是：

- 不能把“好封面”和“好图表页”分别复用
- 不能跨模板抽取最优页面
- 不能做页面级去重

### 3.3 缺少语义搜索和视觉搜索

当前资源库只有源站点清单和基础评分，没有：

- 向量检索
- 页面语义 embedding
- 截图相似度检索
- 页面结构相似度检索

因此还不能根据“新能源汽车行业 + 年度销售 + 科技蓝 + 大量图表”这种需求精准匹配页面。

### 3.4 缺少 Excel 到 PPT 的结构映射

当前表格功能可以生成和处理数据，但还没有把 Excel 结构映射到 PPT 页面类型，例如：

- 月份 + 销售额 -> 折线图页
- 地区 + 指标 -> 地图 / 柱状图页
- 实际值 + 目标值 -> KPI 页

### 3.5 缺少统一设计系统

目前虽然有主题色和版式，但还没有形成真正的 Design System：

- 字体体系
- 字号体系
- 间距体系
- 图表配色
- 图标风格
- 页面节奏

所以生成结果能用，但还不够“像一套完整的专业产品”。

## 4. 当前可保留的优点

- Streamlit 单文件入口，部署门槛低
- 已具备 PPT / 表格双能力
- 已接入 `ppt_template_library` 骨架
- 已能初始化目录、数据库、资源目录和源站点清单
- 已具备后续扩展为页面库、组件库、评分库的落点

## 5. 当前建议的角色划分

### 本地算法适合做

- 文件解析
- PPT 解包与页面抽取
- 页面截图生成
- 哈希去重
- 资源入库
- 元数据写入
- 预览生成
- 评分与排序
- Excel 数据分析
- 图表生成
- 规则标签

### 需要 LLM 参与的部分

- 提示词理解
- 主题 / 行业 / 场景推断
- 页面用途判断
- 内容重写与摘要
- 页面组合建议
- 页序重排建议

### 需要视觉模型参与的部分

- 页面类型识别
- OCR 辅助分类
- 截图质量评分
- 感知去重
- 页面视觉风格识别

## 6. 结论

当前系统已经具备“资源库起点”和“生成器雏形”，但离目标中的“页面级模板库 + 组件级素材库 + 智能组装系统”还有一层结构鸿沟。下一步不是继续堆完整模板，而是先把 PPT 拆成页、拆成组件、拆成可检索对象。
