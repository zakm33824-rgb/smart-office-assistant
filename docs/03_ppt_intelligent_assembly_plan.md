# PPT 智能组装系统开发计划

## 1. 总原则

- 先拆解，再组合
- 先结构化，再智能化
- 先小粒度资源，再整套生成
- 先本地可控，再引入大模型和视觉模型
- 任何阶段都保留回滚能力

## 2. 推荐实施顺序

### Phase 1：现状分析与资源骨架

目标：把当前代码、资源库、目录结构和许可证风险先摸清楚。

交付：

- 现有系统分析报告
- 页面级模板库架构方案
- 智能组装系统开发计划
- 资源源站清单和初始目录结构

### Phase 2：页面拆解与页面分类

目标：把完整 PPT 拆成 slide 级资源。

交付：

- pptx 解包器
- 页面截图生成器
- 页面类型分类器
- 页面 metadata 生成器

### Phase 3：组件提取与组件库

目标：把高质量页面中的可复用元素沉淀下来。

交付：

- 标题组件库
- KPI 组件库
- 图表组件库
- 时间轴 / 流程 / 对比组件库

### Phase 4：搜索与评分系统

目标：让系统能按主题、行业、场景、风格、颜色、页型检索。

交付：

- 标签检索
- 向量检索
- 视觉相似度检索
- 质量评分与去重

### Phase 5：智能组装与设计系统

目标：从不同来源抽取页面后，统一成一套视觉语言。

交付：

- Design System
- 跨模板组装器
- 内容自适应器
- 图表风格统一器

### Phase 6：Excel 到 PPT 映射

目标：让表格数据自动映射成适合的 PPT 页面。

交付：

- Excel 结构分析器
- 指标识别器
- 图表推荐器
- 页面推荐器
- 自动排版器

### Phase 7：反馈与自动优化

目标：记录用户使用行为，持续提高推荐质量。

交付：

- 选择记录
- 删除记录
- 替换记录
- 常用模板统计
- 常用风格统计
- 推荐权重调优

## 3. 当前阶段建议直接做的事情

### 先做本地算法

1. 完整 PPT 拆页
2. 页面分类
3. 页面截图
4. 元数据入库
5. 去重
6. 质量评分
7. 搜索索引

### 再做模型能力

1. LLM 做意图理解
2. 视觉模型做页面识别
3. embedding 做语义检索

### 最后做组装

1. 页面匹配
2. 页面排序
3. 内容适配
4. 视觉统一
5. 导出 PPTX

## 4. 风险点

### 技术风险

- PPT 结构复杂，解包后布局差异很大
- 页面类型识别不稳定
- 视觉去重成本高
- 多来源模板许可证不一致

### 产品风险

- 用户输入过于抽象
- 用户期望与模板实际结构不匹配
- 页面越自由，组装越容易失控

### 控制手段

- 先限制页型集合
- 先限制商用来源
- 先限制 16:9 / 4:3 等标准比例
- 先建立高质量白名单

## 5. 本地 / LLM / 视觉分工

### 本地完成

- 文件读取
- 数据清洗
- 页面拆分
- 统计分析
- 哈希去重
- 预览生成
- 导出

### LLM 完成

- 主题理解
- 场景归纳
- 章节规划
- 内容重写
- 页面组合建议

### 视觉模型完成

- 页面类型识别
- OCR
- 页面风格识别
- 截图质量评分
- 感知相似度判断

## 6. 验收标准

每个阶段至少满足：

- 能运行
- 能回滚
- 能测试
- 能导出
- 能记录日志

PPT 生成侧的最终验收，不是“每页单独好看”，而是：

- 整套 PPT 风格统一
- 页面节奏合理
- 结构和内容匹配
- 视觉语言一致
- 可持续扩充

## 7. 结论

下一步最重要的不是继续堆完整 PPT 模板，而是把“页”与“组件”变成可检索、可评分、可组合的基础资产。只要这个底座稳住，后面的智能组装才有意义。
